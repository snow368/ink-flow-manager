## 01. Product Launch Checklist 2026: 40 Tasks + Free Template [https://dev.to/iris1031/product-launch-checklist-2026-40-tasks-free-template-4n78]

### 核心论点
- 发布成败不在天赋，而在"把产品发布策略写成有序任务清单"，确保凌晨 6 点混乱时不会漏掉关键项。
- 发布分三阶段：**Pre-launch（T-14→T-1，14 天）/ Launch Day（T-0）/ Post-launch（T+1→T+7，7 天）**。
- **策略先于清单**：动手做战术前，先锁定 5 个决策（定位、受众、主渠道、成功指标、必须完美的一个资产）。
- **Pre-launch 决定结果**，Launch Day 主要是执行早已做的决定；**前 3 小时**最吃重，要前置支持者。
- Product Hunt 只是渠道之一，匹配"买家真正聚集的地方"，而非清单最容易执行的地方。

### 可操作战术（带具体做法）

**发布前必锁定的 5 个决策（Strategy First）**
1. 定位：一句话说清"为谁、为什么切换"。说不出一句话，发布日也救不回。
2. 受众：前 100 个用户已经在哪里聚集？答案决定渠道。
3. 主渠道：PH / 社区种子 / 媒体 / 候补名单，选一个主导，其余支持。
4. 成功指标：注册？付费转化？合格 demo？先定义"好发布"长什么样。
5. 必须完美的一件资产：通常是 demo 视频，其余可以 B+，这件不能。

**Phase 1 — Pre-Launch（T-14→T-1）：18 项任务**
- [ ] 写一句话定位，并找 5 个陌生人压力测试
- [ ] 定义理想早期采用者，列出他们聚集地
- [ ] 选主渠道 + 2 个支持渠道
- [ ] 设单一成功指标 + 现实目标
- [ ] 录 60 秒 demo 视频（必须完美的资产）
- [ ] 准备发布物料：图库图、tagline（≤60 字符）、首条评论
- [ ] 为每个渠道写发布日文案（按平台定制）
- [ ] 搭建/刷新落地页，含清晰 CTA + 社会证明
- [ ] 若上 PH：锁定 hunter 并 brief（参考 how-to-pick-a-hunter）
- [ ] 提前写好 maker comment（参考模板）
- [ ] 预热受众：剧透发布、建 notify 名单
- [ ] 排定 10–20 个活跃支持者，首小时到场
- [ ] 给核心人脉准备 DM/邮件（个性化，非群发）
- [ ] 设分析 + 专属渠道群，归因发布流量
- [ ] 起草 FAQ + 异议处理回复，可快速粘贴
- [ ] 排定发布日期/时间（避开假日与强竞争发布）
- [ ] 全链路 dry run 注册→价值流程，清除每个死路
- [ ] 准备发布后留存邮件/引导序列

**Phase 2 — Launch Day（T-0）：12 项任务**
- [ ] 在主渠道最优时间上线
- [ ] 立即发准备好的 maker comment
- [ ] 通知预热支持者——先、且个性化
- [ ] 首小时内发核心人脉 DM/邮件
- [ ] 按平台定制的文案发支持渠道
- [ ] 几分钟内置回**每一条**评论——个性化，非模板
- [ ] 监控问题与异议，粘贴准备好的答案
- [ ] 实时看分析，记录哪个渠道真转化
- [ ] 支持者到场时公开致谢（保持势能可见）
- [ ] 首小时后仍保持能量，别发一次就消失
- [ ] 注意力高峰时抓取邮件/leads
- [ ] 截图里程碑，作发布后社会证明

**Phase 3 — Post-Launch（T+1→T+7）：10 项任务**
- [ ] 给每个新注册发引导/留存序列
- [ ] 个性化跟进最高意向 leads
- [ ] 写发布复盘（数字 + 教训）给受众
- [ ] 把发布日社会证明转成落地页资产
- [ ] 分析"带来留存用户"的渠道，而非只看注册
- [ ] 找 5 个新用户做快速访谈，找激活缺口
- [ ] 修复注册→价值流程中最大流失点
- [ ] 经营关系（hunter、支持者、评论者）
- [ ] 把发布内容改造成 evergreen 搜索文章
- [ ] 决定下一次发布节奏——没有跟进势能会死

### 关键数字（标 ★）
- 30+ 次 Product Hunt 日榜 #1（作者协助达成）★SELF-REPORTED
- 数据区间：2022–2026 自有组合 ★SELF-REPORTED
- 胜利通常发生在 **14 天** pre-launch 窗口
- 决定结果的是上线**前 3 小时**（早期势能复利）
- 发布三阶段：pre(14天) / day / post(7天)
- 最适合 dev tools/maker/B2B 的渠道 = Product Hunt；consumer/niche = 社区种子（Reddit、Discord）
- 单句 tagline ≤60 字符；demo 视频 60 秒
- 支持者规模 10–20 人；压力测试找 5 个陌生人
- 作者为 ex-AFFiNE COO，辅导 150+ AI 创业公司 GTM ★SELF-REPORTED

### 模板/清单（结构化保留）
- 上述 40 项任务清单（18 pre + 12 day + 10 post）即核心 SOP。
- 5 决策锁定表（定位/受众/渠道/指标/必须完美资产）。
- 配套作者开源 AI skill：`npx skills add Gingiris-1031/gingiris-skills`（MIT 许可，45+ 增长/发布/SEO-GEO skill）。

### takeaway
清单执行的是策略，不是替代策略。把发布当成"前 14 天做功课、当天执行、后 7 天复利"的三段式系统，而非一天的热闹。那张餐巾纸（4 项）和 #1 的差距，从不在才华，而在"按顺序写下来的任务清单"。

## 02. Product Hunt Launch: 7 Pre-Launch Checks for 2026 [https://dev.to/iris1031/product-hunt-launch-7-pre-launch-checks-for-2026-2ii4]

### 核心论点
- 强发布赢在"准备质量"，而非当天噪音。表现差通常因信息太宽、证明太薄、下一步不清——藏在 prep 里。
- 最重要的是三大 pre-launch check：定位清晰、证明充分、单一清晰 CTA。
- maker comment、截图、落地页文案应强化同一承诺（一致性强于各自精彩）。

### 可操作战术（带具体做法）
1. **检查标题是否点出真实结果**：能否命名具体受众/工作流？指向一个锐利结果？避开内部行话？在 PH 之外仍讲得通？
2. **检查截图 1 是否快速证明价值**：展示核心工作流/产出；让品类一眼可辨；含一个可见证明信号；去掉干扰。
3. **检查 maker comment 是否打开清晰路径**：含 4 块——痛点（为何造）、用户（谁最获益）、证明（结果/牵引信号/工作流示例）、下一步（试/star/预约/回复）。
4. **检查落地页是否只有单一主 CTA**：预警信号——首屏多个主行动互相竞争；证明不足就放 CTA；不说明产品给谁；移动端要滚太久才到行动。
5. **检查是否准备好异议答案**：提前写清——为何不用现有工具、谁最合适、是否只是套壳、需要什么搭建/团队。
6. **检查发布流量能否转成漏斗**：提前备——引导邮件/欢迎流、短 demo/证明视频、FAQ/对比页、创始人复盘、再营销/分层列表。
7. **检查发布后证明能否复用**：立即复用——评论里 top 问题、最强一句话反馈、反响最好的截图、maker 回复里的证明片段、按来源统计的注册/demo。

### 关键数字（标 ★）
- 无独立统计数字；战绩声明沿用 30+ PH #1 ★SELF-REPORTED（见文末 bio 及系列其他篇）。

### 模板/清单
- **发布就绪快速审计表（弱信号 vs 强信号）**：
  | 维度 | 弱信号 | 强信号 |
  |---|---|---|
  | 标题 | 宽泛聪明 | 具体受众 + 结果 |
  | 截图1 | 漂亮 UI | 可见工作流 + 证明 |
  | maker comment | 仅致谢 | 痛点/证明/CTA |
  | 落地页 | 多下一步 | 一个明显动作 |
  | 跟进 | 无系统 | 准备好的复用计划 |

### takeaway
本周若要改善一个发布：收紧标题、围绕可见证明重建截图 1、简化点击后第一个 CTA。这三项 pre-launch check 通常比多追几个当天票带来更多转化提升。

## 03. Product Hunt Launch: 7 Conversion Loops for 2026 [https://dev.to/iris1031/product-hunt-launch-7-conversion-loops-for-2026-3ac8]

### 核心论点
- 发布结果很少只靠当天流量。赢家在 midnight PT 前就建好转化环：更紧定位、更清证明、更快 maker 回复、upvote 后一个明显下一步。
- 页面有注意力但没 demo/注册/star，问题通常是 reach，而是 message-to-action fit（信息到行动的匹配）。
- 优化 upvote 而非下一步意图，是最大误区。

### 可操作战术（带具体做法）
1. **标题匹配一个锐利用户结果**：陌生人 5 秒看懂；像用户问题而非内部行话；一个具体结果而非三个半承诺；同样标题也能用在落地页。
2. **首张截图承载真实证明**：结果导向标题；一个可见工作流或前后对比；一个指标/证言/社会证明；最小视觉噪音。
3. **写打开转化路径的 maker comment**：含触发（市场/工作流什么变了）→用户（谁最痛）→证明（结果/牵引/示例）→下一步（demo/repo/候补/免费工具）。
4. **减少落地页 CTA 冲突**：预警——book demo/start free/join Discord/star GitHub 同时出现；证明不足就求承诺；首屏藏了品类；移动端滚太久。每类受众选一个主行动，其余支持。
5. **建前 6 小时评论环**：几分钟内置回；把常见问题变成迷你证明点；加截图/示例/短 demo；收集重复异议更新落地页。
6. **立即重打包发布日证明**：复用 top 评论、最佳截图、最有说服力的一句话反馈、暴露购买意图的问题、按来源转化数据——用于邮件/引导/未来发布。
7. **发布前就设计发布后那一周**：准备——复盘帖、带教训的创始人笔记、案例式 thread/文章、给发布注册用户的引导序列、给参与未转化用户的社区跟进。

### 关键数字（标 ★）
- 战绩：30+ PH #1 ★SELF-REPORTED；前 6 小时评论环为关键窗口。

### 模板/清单
- **发布审计表（弱/强信号）**：同 #02 结构（标题/截图1/maker comment/落地页/发布后）维度一致。
- **前 6 小时评论环 SOP**：快速回复 → 问题转证明点 → 补素材 → 收集异议。

### takeaway
本周改一个发布：收紧标题、围绕证明重建首图、简化点击后 CTA 路径。这三处改动通常比制造更多当天噪音更快见效。

## 04. Product Hunt Launch: 7 Maker Comment Loops for 2026 [https://dev.to/iris1031/product-hunt-launch-7-maker-comment-loops-for-2026-1lgk]

### 核心论点
- 强发布不止于标题/缩略图/首波 upvote。2026 赢家把 maker comment 当"转化面"：澄清定位、快速答异议、首波流量后保持故事推进。
- 评论质量实时塑造访客如何理解产品；强评论环还能产出可复用的引导/SEO/社区文案。

### 可操作战术（带具体做法）
1. **用一个清晰工作流开场，而非公司传记**：结构——为谁（命名第一段受众）→做什么活（一个工作流，非整个路线图）→为何现在（什么变了/发了/新变得有用）。
2. **把重复问题变成短转化资产**：提前备——和明显替代品的差异？谁最合适？注册后发生什么？搭建要多久？答案就绪则线程冷静且有说服力。
3. **用证明而非更多宣称答异议**：有用格式——一张截图/gif（展示所述工作流）、一条用户原话（同用例）、一个指标/前后结果（具体数字比泛泛夸赞传得更远）。
4. **保持评论线程与页面定位对齐**：检查——首条回复是否强化与标题同用例；后续答案是否复用同核心措辞；每个证明点是否支撑主承诺；是否意外在回复里扩大了品类。
5. **用 maker comment 桥接发布后引导**：保存——用户自然重复的原话、持续阻挡注册的异议、赢得最强回复的示例、让产品"秒懂"的解释。
6. **当天内建迷你环，而非一次大爆发**：节奏——早段（钉首条 maker 故事+快速答显性问题）；中段（首波流量平息后补新鲜证明/示例/工作流）；晚段（给晚来者一个更深洞察/模板/用户故事）。
7. **把最佳评论语言复用到其他渠道**：落地页 FAQ、同关键词簇博客、社区帖/创始人 thread、引导文案/生命周期邮件、应用商店截图/副标题。

### 关键数字（标 ★）
- 无新增硬性数字；系列战绩 30+ #1 ★SELF-REPORTED。

### 模板/清单
- **maker comment 实操清单**
  - 发布前：起草聚焦单一工作流的开场；备 3 个可能异议的答案；收集 1 张截图 + 1 句原话 + 1 个具体结果；对齐评论措辞与标题/tagline；指定前几小时谁回复。
  - 发布日：早发 maker 介绍；用证明而非仅解释答重复问题；保存用户强措辞备复用；补第二波带新鲜示例的评论；记录应改进引导/落地页的异议。

### takeaway
更好发布来自把 maker comment 当产品故事的一部分，而非发布后清理。本周改一处：重写首条 maker comment，让它在一分钟内讲清一个工作流、一类受众、一个证明点——常同时拉升转化质量与整场对话价值。

## 05. Product Hunt Launch: 7 Positioning Fixes for 2026 [https://dev.to/iris1031/product-hunt-launch-7-positioning-fixes-for-2026-41b0]

### 核心论点
- 更好发布始于更锐定位，而非更大声推广。2026 赢家收缩品类、以一个清晰用例领跑、在流量到达前消除混淆。
- 故事模糊时，更多曝光只是把漏洞放大。

### 可操作战术（带具体做法）
1. **别以最宽品类领跑**：收紧——标题下第一行、品类标签、你想让人先想到的那个工作流、想先赢的受众段。能做的事有十件，页面仍应只以一个难忘的门领跑。
2. **把承诺改成听起来像结果**：从能力→结果（说用户什么变了，而非产品含什么）；从泛价值→具体工作流（具体用例在评论/转发里传更远）；从野心→即时性（展示今天能做什么，而非路线图未来解锁）。
3. **让英雄证明匹配主主张**：证明资产——一个前后工作流、一条带可见产出的短 demo、一条绑定该用例的用户/客户原话、一个快速降低风险的可信信号。主张是速度就秀速度，是协作就秀协作。
4. **发布日前消除品类混淆**：信号——人们拿你错类比；评论问"这到底是什么"；用户复述错标题；预览里同一异议反复出现。这是定位问题不是推广问题。
5. **保持发布文案与引导文案对齐**：检查——首次运行体验是否匹配发布页承诺；首个 CTA 是否延续同用例；首屏是否快速确认价值故事；邮件跟进是否重复同定位。
6. **围绕定位建可信资产**：公开文档/repo 链接、透明示例/模板、答重复异议的 FAQ、与新工作流对比的简单对照。
7. **把最佳发布定位复用到其他渠道**：落地页 hero、同关键词簇博客、社区帖/创作者触达、引导消息、应用商店文案。

### 关键数字（标 ★）
- 系列战绩 30+ #1 ★SELF-REPORTED；无新增独统计。

### 模板/清单
- **定位实操清单**
  - 发布前：让 5 个用户用一句话复述产品；选识别最快的单一用例；围绕结果而非功能重写标题；把最强匹配证明放顶部；让引导延续同故事。
  - 发布日：maker comment 聚焦一个工作流；用同定位语言答异议；观察人们自然重复哪句措辞；保存强措辞供未来 SEO/生命周期内容。

### takeaway
更强发布来自流量到达前修定位。本周选一个动作：把页面收窄到一个用例、一个承诺、一个互相支撑的证明资产——常是"嘈杂发布"与"令人难忘发布"的分界。

## 06. Product Hunt Launch: 7 Pre-Launch Signals for 2026 [https://dev.to/iris1031/product-hunt-launch-7-pre-launch-signals-for-2026-44ck]

### 核心论点
- 强发布始于发布日前验证信号：哪类受众最先响应、哪段 demo 被重播、哪类异议重复、哪句信息人们无需帮助就能复述。
- 信号弱就上更多流量，通常只是更快暴露问题。把发布日当测试太晚——要在流量到达前就知道哪个故事成立。

### 可操作战术（带具体做法）
1. **检查人们能否正确复述定位**：让 5 个友好用户看短 demo 后描述产品；对比其措辞与你的标题；注意过度泛化或误解品类处；收紧页面直到复述措辞一致。
2. **衡量哪个用例识别最快**：信号——快速点头（无需解释就懂一个工作流）；更好后续问题（不问"做什么"而问"怎么融入我的工作流"）；更强转介绍语言（知道还该给谁看）。
3. **在公开前追踪异议**：提前记录——定价困惑、可信/背书怀疑、迁移摩擦、品类混淆、工作流匹配问题。重复异议=文案方向。
4. **寻找 demo 重播与分享信号**：比点赞更重要的——哪段被重播最多、人们事后提到哪个时间戳、哪张截图/gif 被内部转发、哪个 CTA 换来真实回复而非礼貌反应。这些揭示 hero 区该强调什么。
5. **区分好奇与真实拉动（pull）**：真实 pull——用户问"何时发布"（时机兴趣非礼貌）；潜在客户问"工作流是否支持我团队"（购买/采用意向）；有人转发给同事/朋友（价值故事可携带）。
6. **审计可信资产是否就绪**：备——能答技术问题的 repo/文档链接、短证明故事/用例示例、常见异议的 crisp FAQ、注册后简单引导路径。
7. **到处复用最佳 pre-launch 语言**：PH tagline 与 maker comment、落地页 hero、引导首跑文案、后续博客、创作者/社区触达。

### 关键数字（标 ★）
- 系列战绩 30+ #1 ★SELF-REPORTED；建议让 5 个用户做复述测试；pre-launch 信号在"发布前一周 / 发布前一天"两个节点检查。

### 模板/清单

  - 发布后：用更清语言更新落地页；发一篇复盘 + 一篇异议处理文；在邮件/demo 流复用发布学习；把持久兴趣路由进自有渠道。

### takeaway
发布最佳表现当它像连通系统而非独立活动。清晰信息匹配、更快激活、更早证明、更好 CTA、 disciplined 发布后复用，通常比喧闹的发布日表演更重要。漏斗紧，同一波注意力能在注册/搜索/社区信任/甚至 GitHub 发现里长期回报。

## 12. Product Hunt Launch: 7 Positioning Checks for 2026 [https://dev.to/iris1031/product-hunt-launch-7-positioning-checks-for-2026-19p1]

### 核心论点
- 强发布外面看像流量问题，团队内常是定位问题。陌生人落地页若说不清为谁/为何现在重要/有何不同，再准时发布也变礼貌好奇而非注册。
- 2026 转化最好的团队不只收 upvote，而是让页面几秒内可读（legible）。把定位检查当 preflight check。

### 可操作战术（带具体做法）
1. **陌生人 5 秒内能否识别用户？** 别问"tagline 听起来聪明吗"，问"对的用户能否瞬间认出自己？"例：太宽"AI workspace for modern teams"；更好"AI research workspace for product teams writing weekly briefs"；更好"Meeting assistant for founders who hate manual CRM updates"。目标不是取悦所有人，是让对的人停止滑动。
2. **结果是否在功能列表前清晰？** 先让明显——什么痛苦任务被移除、什么结果更快出现、注册后什么工作流改变。页面应让人想"哦，这是帮我做什么的"再评估怎么做到的。
3. **差异是否重要？** "更快/更简单/AI 驱动"不够，词太廉价。更强角度——特定受众专精（不怕说只为谁）；工作流楔入（替代一个痛苦步骤而非整栈，更易信）；价值速度（3 分钟而非 30 分钟到价值）；信任/证明（开源可信/公开模板/客户证据降怀疑）。
4. **图库是否讲一个故事而非七个半故事？** 更好顺序——图1 解释主承诺；图2 展示真实工作流中的产品；图3 证明一个差异点；图4 处理一个可能异议。每帧推进同核心故事，若每帧引入新故事则都不落地。
5. **maker comment 是否深化定位？** 它能——解释为何这问题现在值得解、秀一个锐利用例、揭示一个难产品权衡、求能引来深思回复的反馈。接地气的 maker comment 常比过度抛光发布文案转化好。
6. **点击后路径是否匹配发布承诺？** 自服 SaaS 送聚焦注册路径（一个明显激活步）；B2B SaaS 送高意向流量去 demo/用例页/ROI 解释；开发者产品路由好奇访客去文档/GitHub/模板/公开示例项目。
7. **发布信息能否活过一天？** 最佳检查：PH 定位能否也驱动——首页 hero、引导文案、创始人复盘、SEO 文章跟进、销售/demo 开场。若信息只在 PH 有效，可能太活动专属；强定位跨渠道旅行好。

### 关键数字（标 ★）
- 关键时效：**5 秒内**识别用户（作者经验）；价值速度对比"3 分钟 vs 30 分钟"（示例数字）；系列战绩 30+ #1 ★SELF-REPORTED。

### 模板/清单
- **定位检查清单**
  - 发布日前：重写 tagline 直到用户明显；结果在功能列表前可见；定义一个可信差异点；把截图故事收紧成一个叙事；起草增信非噪音的 maker comment；确保落地页匹配发布承诺。
  - 发布中：看人们重复哪句措辞；记什么最快让访客困惑；若出现模式就精修落地页；在回复里复用强受众语言。
  - 发布后：把重复问题变 FAQ 文案；信号新鲜时发复盘；从真实异议发 SEO 跟进内容；把最佳定位句留在自有表面。

### takeaway
发布失败不只因 timing 或分发弱，常因冷流量到达时产品故事仍模糊。若收紧为谁/创何结果/为何不同/访客该去哪，整场发布更易转化。

### 可操作战术（带具体做法）—— 7 标准（按优先级）
1. **主题契合 > 粉丝数**：一个每周发 AI dev 工具、1万粉的 hunter，比 20万粉发烹饪 app 的 hunter 对你的 AI dev 工具更有用。PH 算法更重"品类相关"用户的 upvote；hunter 受众是品类相关性的代理。通用 hunter 受众 80% 无关噪音，niche hunter 受众转化高 3–5 倍。**做法**：搜 hunter 最近 20 次 hunt，若 <70% 在你品类，跳过——无论粉丝多少。
2. **近 90 天活跃**：2023 发 10 次、2026 发 0 次的 hunter 是休眠的，粉丝已取关/退订，"触达"是理论值。**做法**：看 profile→近期 hunts，过去 90 天至少 3 次；加分项：那些近期 hunt 的 upvote 应在 200–800 区间最低（否则 hunter 已丢互动信号）。
3. **hunter 真用你的产品**：这是"业余创始人"与"有经验 PH 创始人"间最大护城河。真用的 hunter 写更好发布评论、当天真实回复、最关键——信到愿意转推/交叉发。**做法**：发布前 4–6 周给 hunter 免费 Pro 账号，观察是否真用；若 2 周内没打开，是"纸上 hunter"，换。
4. **时区对齐**：PH 12:01 AM PST 上线，前 2 小时评论在算法里权重最高。太平洋 hunter 上线后 30–60 分钟能评，欧洲 hunter 在睡。**做法**：PT 或 ET hunter > 欧洲 > 亚洲。亚洲创始人（作者在昆山）这是招募西方 hunter 而非自提交的最大原因。
5. **曾对真实创始人说 yes，非只名人**：顶级 hunter 每周收 20–40 个请求，大多不回。10K–30K 粉层的 hunter 对无关系的创始人响应度高得多。**做法**：看 hunter 最近 5 次 hunt，若都来自 YC 系或 50K+ Twitter 粉，你的 DM 会被忽略；找近期 hunt 含几个 <10K 粉创始人的——那才是现实可求的人。
6. **在受众所在的其它平台也在**：hunter 的 PH 粉丝是一个渠道，其 Twitter/LinkedIn/dev.to 存在放大发布日。**做法**：选 Twitter/X 受众也相关的 hunter；5K 相关 Twitter 粉 > 50K 随机 PH 粉。
7. **Day 1 后会留下来**：最佳 hunter 回复评论者 72 小时，最差 12:01 AM 提交就消失。**做法**：直接问"你能回复评论到周四吗？"其回答说明一切；若含糊，换人。

- **情形1**：已做 6 周发布前社区建设（私 Discord/Slack 有 100+ 活跃成员，且发布前亲自私信 200+ 相关创始人）→ hunter 边际贡献小，你已直接建起 upvote 速度。

## 14. Product Hunt Launch: 8 Prep Moves That Compound [https://dev.to/iris1031/product-hunt-launch-8-prep-moves-that-compound-eon]

### 核心论点
- 强发布通常在发布日前已定。创始人常聚焦 hunter/upvote/完美发帖时间，但真正杠杆在定位、暖流量、引导就绪、快反馈环。
- 准备好在四层创造价值：早期采用者注意力 → 转注册/demo/repo 访问 → 评论异议的语言洞察 → 复利进 SEO/GEO 的后续内容。准备击败表演。

### 可操作战术（带具体做法）
1. **动图库前先修定位**：页面几秒内答 5 问——是什么、为谁、为何现在不同、创何结果、下一步做什么。价值主张模糊，强流量也漏。
2. **写带用户与结果的 tagline**：公式——品类/产品类型 + 目标用户 + 具体结果。例"AI workspace for teams"太宽；"AI research copilot for product teams shipping weekly briefs"给人真实可评估的东西。
3. **页面上线前暖出真实流量**：首小时有真实兴趣而非沉默，感觉很不同。暖流量源——现有用户、早已知产品的创始人朋友、通讯订阅者、你真帮过的社区成员、相邻产品用户。警惕把 PH 当纯冷获取；暖需求让页面早显活力。
4. **把 maker comment 当转化资产**：强 maker comment 含——让你去造的问题、一个具体工作流/用例、一件难做对的事、一个有用的反馈请求。好 maker comment 让团队显锐利可信，非过度抛光。
5. **为已知异议备回复**：多数发布遇可预测问题（定价、差异、集成、引导、目标用户）。建轻量回复库：为谁最佳/不为谁/与替代品对比/前 10 分钟长啥样/如何给产品反馈。快而 thoughtful 的回复保持线程有用活跃。
6. **对齐落地页与发布承诺**：点击落通用首页则表现差。更好路径——聚焦注册流、高意向买家的 demo 请求、技术用户的文档/用例页、开发者产品的 GitHub repo、带明确加入理由的候补。
7. **把发布连到收入动作**：SaaS 发布页不该停在认知，应支撑更广管线（自服激活、demo 请求、团队扩张、内容再营销）。
8. **还有注意力时就规划发布后内容**：发布制造反馈与语言爆发，趁新鲜抓。易做后续——发布复盘、基于重复问题的 FAQ、对常见替代品的对比页、创始人教训 thread、基于评论困惑的引导微调。

### 关键数字（标 ★）
- 系列战绩 30+ #1 ★SELF-REPORTED（文末 bio：4 年联合创立 AFFiNE 60K+ stars、30x PH #1、现 bootstrap Analook）；作者 playbook 集 90+ 篇。

### 模板/清单
- **精简发布清单**
  - 发布前一周：收紧定位与 tagline；为清晰重写首屏；起草 maker comment；用语境 brief 暖支持者；检查分析/引导/demo 流。
  - 发布日：评论保持活跃；把流量路由到聚焦下一步；记重复异议与问题；跨渠道复用高信号语言。
  - 发布后一周：发复盘/事后；基于反馈更新落地页；把评论主题变 SEO 内容；跟进最高意向访客。

### takeaway
发布最佳表现当它被设计为更广系统的一部分。清晰定位、暖流量、thoughtful maker comment、快回复、聚焦点击后路径，仍胜喧闹战术。拿到真价值的团队，通常为转化与跟进准备，而非仅为可见度。

## 15. Product Hunt Launch Checklist: 9 Plays for 2026 [https://dev.to/iris1031/product-hunt-launch-checklist-9-plays-for-2026-4a1i]

### 核心论点
- 好发布 2026 仍有杠杆，但仅当发布页/暖流量/maker 回复/发布后内容协同。太多团队把 PH 当一日彩票；更好做法是把它当喂给 SEO/注册/客户访谈/甚至 GitHub 发现的紧凑分发事件。
- 若发布策略停在 upvote，你留了大部分价值在桌上。

### 可操作战术（带具体做法）
1. **5 秒内让定位可懂**：首屏答——是什么、为谁、为何现在不同、访客该做什么。若访客需解码你的品类，页面已在漏能量。
2. **写具体而非聪明的 tagline**：成分——品类清晰 + 具体用户 + 清晰收益。例"AI workspace for modern teams" vs "AI research copilot for product managers who need weekly briefs faster"（后者更窄但给人真实反应点）。
3. **用真人在前几小时暖场**：冷页显风险，暖页显活力。暖源——现有用户、通讯订阅者、maker 朋友与创始人同侪、你已贡献的社区、相邻工具用户。非造假势能，是别让发布首段显空。
4. **把 maker comment 当迷你案例研究**：含——让你去造的问题、一个具体工作流/用例、一条建造硬教训、一个清晰反馈请求。教东西的评论比泛泛庆祝文案常被忽略好得多。
5. **前 12 小时快回**：快回复是真增长杠杆——公开答异议、显团队在场、给后来访客更多留页理由、浮现可复用进 FAQ/落地页的语言。最被低估的发布习惯之一。
6. **把 PH 流量指向自有资产**：点击后送弱首页无下一步则流量尖峰速褪。更好目的地——邮件捕获、引导流、候补/demo CTA、文档/用例页、开发者产品的 GitHub repo。
7. **让发布匹配收入动作**：非每个 PH 受众同方式转化。卖向团队，发布页与落地页应暗示真实扩张路径（自服注册/demo/模板/免费工具皆可用，但需匹配商业模式）。
8. **把发布反馈变搜索内容**：发布不随排名定而终。易做后续——发布复盘、客户异议变 FAQ、对比页、教训 thread、基于问题的功能深潜。一次发布变多个搜索入口而非一天噪音。
9. **比你想的更早本地化分发**：很多全球团队英文发完就停。页面本身英文 OK，但分发不该单市场（若别处也有需求）。轻量本地化——在区域社区发复盘、本地化落地页标题、移动产品本地化截图文案、请不同市场 power user 把价值主张译成本土自然语言。

### 关键数字（标 ★）
- 关键时效：**前 12 小时**快回（作者经验）；系列战绩 30+ #1 ★SELF-REPORTED。

### 模板/清单
- **9 Plays 发布清单**
  - 发布前一周：收紧定位与 tagline；备截图与图库顺序；写 maker comment 草稿；确认分析与引导工作；用语境而非垃圾请求 brief 暖支持者。
  - 发布日：以清晰首屏价值发；评论保持活跃；分享到 2–3 个暖渠道；收集重复问题与异议。
  - 发布后一周：发复盘；基于评论更新落地页；把问题变 FAQ/博客内容；把最佳角度复用进 SEO 文与社交帖。

### takeaway
强发布少在玩平台，多在协调注意力。清晰定位、暖早期流量、快回复、disciplined 发布后跟进，仍胜更响战术。从 PH 拿到最多的团队，通常把发布日当内容与转化环的起点，而非终点。

## 对 ink-flow-manager 项目的可借鉴点

> ink-flow-manager = SaaS 产品 InkFlow（AI 写作/内容流管理）。未必上 Product Hunt，但"发布前定位 / pre-launch signals / proof loops"思路可直接映射到**内容发布流程**与 **GSC 上线清单**。

1. **把"发布前 5 决策"变成内容发布前清单**：每篇要发的文章/功能更新，先锁定——一句话定位（为谁、解决什么写作痛点）、目标受众聚集地（哪个社群/关键词）、主分发渠道（GSC 自然搜索 / 邮件 / 社区）、成功指标（GSC 展现/点击、注册激活）、必须完美的那一件资产（首屏价值主张或 demo）。避免"写了就发"。
2. **用 pre-launch signals 替代"发了看数据"**：文章上线前先找 5 个目标用户短暂预览，让他们用一句话复述"这文章帮我解决什么"；若复述不一致，先改标题/导语再发。把"人们自然重复的那句措辞"直接用作 GSC 的 Title/Description 与首段，提升搜索相关性。
3. **建"proof loop"进 GSC 上线清单**：上线后 48 小时内监控 GSC 的展现/点击/CTR，把高印象低点击页面（=证明/标题不匹配）当"评论区异议"处理——重写 snippet、补首屏证明（用例截图、前后对比、用户原话）；把高点击但高跳出页当"激活缺口"修首屏价值。
4. **把"发布后内容环"固化成节拍**：一次大功能/内容发布后，48 小时内发复盘、把评论/搜索词里的真实异议写成 FAQ 或长尾文章，让单点发布变多个 GSC 搜索入口，而非一天噪音。
5. **用"激活环"看 InkFlow 自身留存**：别只盯注册，定义"首次成功事件"（如生成第一篇达标草稿），缩短到首次价值时间（目标 <5 分钟），把发布/搜索来的流量路由进聚焦引导流而非通用首页——这正是文中所说"漏斗在点击后漏，而非流量少"。

> 一句话：把 Iris 的"三段式发布系统（前14天做功课 / 当天执行 / 后7天复利）"压缩成 InkFlow 的"内容/GSC 上线 SOP"——前做功课（定位+信号测试）、发时执行（清晰首屏+快响应）、发后复利（复盘+复用语言+修激活）。
